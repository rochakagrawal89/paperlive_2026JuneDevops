Resolving a Git Merge Conflict

A guided, step-by-step lab that walks you through creating, identifying, and resolving a real merge conflict — with explanation of every concept along the way.

Difficulty: Intermediate Duration: 45–60 min Prereq: Git basics, branching Tool: Terminal + any text editor
Phases
4 stages
Total Steps
18 sub-steps
Skills covered
Create · Detect · Resolve · Verify
Bonus challenge
VS Code merge editor
Learning objective

By the end of this assignment you will be able to explain why merge conflicts happen, identify conflict markers in a file, and confidently resolve conflicts using both the command line and a visual editor — skills you will use every time two developers edit the same part of a file.

Background concept
💡
A merge conflict occurs when two branches have changed the same lines in the same file and Git cannot automatically decide which version to keep. Git pauses the merge and asks you — the developer — to make the call. This is a normal, expected part of collaborative development.

1

      

        
Phase 1 · Setup

        
Create and initialise the project repository

        
Set up a fresh Git repo with an initial commit to give branches a common ancestor.

      

      
›

    

    


      

        
1a

        

          
Create a working directory and initialise Git

          
We start with a completely empty project so you can see every commit that leads to the conflict.

          

            $ mkdir git-conflict-lab && cd git-conflict-lab

            $ git init

            Initialized empty Git repository in .../git-conflict-lab/.git/
          

        

      


      

        
1b

        

          
Create the shared file and make the first commit

          
This is the common ancestor commit. Both branches will start from here. The conflict arises because each branch edits the same line after this point.

          

            
📄 app.py

            

              greeting = "Hello, World!"
print(greeting)
            

          

          

            $ echo 'greeting = "Hello, World!"' > app.py

            $ echo 'print(greeting)' >> app.py

            $ git add app.py

            $ git commit -m "initial: add greeting script"

            [main (root-commit) a1b2c3d] initial: add greeting script
          

          

            ✓
            Run git log --oneline. You should see exactly one commit labelled "initial: add greeting script".
          

        

      


    

  


  
  

    

      
2

      

        
Phase 2 · Diverge

        
Create two branches that edit the same line

        
Simulate two developers working on different changes to the same file simultaneously.

      

      
›

    

    


      

        
2a

        

          
Create branch feature/alice and change the greeting

          
Alice is adding a personalised greeting. She checks out a new branch from main and edits line 1.

          

            $ git checkout -b feature/alice

            Switched to a new branch 'feature/alice'
          

          
Now open app.py in your editor and change line 1 to:

          

            
📄 app.py (after Alice's edit)

            

              greeting = "Hello from Alice!"

              print(greeting)
            

          

          

            $ git add app.py

            $ git commit -m "feat: personalise greeting for Alice"

            [feature/alice d4e5f6a] feat: personalise greeting for Alice
          

        

      


      

        
2b

        

          
Switch back to main and create branch feature/bob

          
We switch back to main first — this ensures Bob's branch is based on the original file, not Alice's changes.

          

            $ git checkout main

            Switched to branch 'main'

            $ git checkout -b feature/bob

            Switched to a new branch 'feature/bob'
          

          

            💬
            
Open app.py right now and confirm you still see "Hello, World!" — that proves Bob is starting from the same base as Alice.

          

        

      


      

        
2c

        

          
Make Bob's change on the same line and commit

          
Bob also changes line 1, but with a different string. This is the exact setup that will cause a conflict when we try to merge both branches into main.

          

            
📄 app.py (after Bob's edit)

            

              greeting = "Hello from Bob!"

              print(greeting)
            

          

          

            $ git add app.py

            $ git commit -m "feat: personalise greeting for Bob"

            [feature/bob 7g8h9i0] feat: personalise greeting for Bob
          

          

            ✓
            Run git log --oneline --all --graph. You should see three commits: the initial one plus one commit on each branch, both forking from main.
          

        

      


    

  


  
  

    

      
3

      

        
Phase 3 · Conflict

        
Trigger the merge conflict and understand what Git is telling you

        
Merge both branches into main and read every part of the conflict output.

      

      
›

    

    


      

        
3a

        

          
Merge Alice's branch into main — this succeeds cleanly

          
The first merge always succeeds because there is nothing to conflict with; Alice's branch is simply ahead of main at this point (fast-forward).

          

            $ git checkout main

            $ git merge feature/alice

            Updating a1b2c3d..d4e5f6a
Fast-forward
 app.py | 2 +-
 1 file changed, 1 insertion(+), 1 deletion(-)
          

        

      


      

        
3b

        

          
Merge Bob's branch into main — this produces a conflict

          
Now main already contains Alice's version of line 1. Bob's branch also changes line 1. Git cannot decide which version to keep — so it stops and asks you.

          

            $ git merge feature/bob

            Auto-merging app.py
CONFLICT (content): Merge conflict in app.py
Automatic merge failed; fix conflicts and then commit the result.
          

          

            ⚠️
            
Do not panic. Git has not broken anything. Your repository is in a MERGING state — all changes are preserved and safe. Git is simply waiting for your decision.

          

        

      


      

        
3c

        

          
Inspect the current state with git status

          
Before editing anything, read what Git is telling you. The output clearly identifies which files have conflicts and what actions are available.

          

            $ git status

            On branch main
You have unmerged paths.
  (fix conflicts and run "git commit")
  (use "git merge --abort" to abort the merge)

Unmerged paths:
  (use "git add <file>..." to mark resolution)
	both modified:   app.py

no changes added to commit (use "git add" and/or "git commit -a")
          

          

            💡
            
"both modified" is Git's way of saying: both branches touched this file at the same location. You can always run git merge --abort to completely undo the merge attempt and return to the pre-merge state.

          

        

      


      

        
3d

        

          
Open app.py and read the conflict markers

          
Git has rewritten app.py with special markers to show you exactly what both sides changed. You need to understand each part before editing.

          

            
📄 app.py (with conflict markers)

            

              <<<<<<< HEAD

              greeting = "Hello from Alice!"

              =======

              greeting = "Hello from Bob!"

              >>>>>>> feature/bob

              print(greeting)
            

          

          

            

              

                <<<<<<< HEAD
                Start of your current branch's version (main, now containing Alice's change)
              

              

                =======
                The dividing line between the two conflicting versions
              

              

                >>>>>>> feature/bob
                Start of the incoming branch's version (Bob's change)
              

            

          

        

      


    

  


  
  

    

      
4

      

        
Phase 4 · Resolution

        
Resolve the conflict, stage the fix, and complete the merge

        
Edit the file to produce the final agreed-upon version, then finish the merge commit.

      

      
›

    

    


      

        
4a

        

          
Decide on the resolution strategy

          
You have three common options. Each is valid in different real-world situations. For this lab, use option C (combine both).

          

            

              Option A
Keep your branch (HEAD). Delete Bob's lines entirely.
            

            

              Option B
Keep incoming (theirs). Delete Alice's lines entirely.
            

            

              Option C ✓
Merge both changes into a single new line.
            

          

        

      


      

        
4b

        

          
Edit app.py — remove all conflict markers and write the resolved version

          
Open the file in your editor. Delete every line that starts with <<<, ===, or >>>. Write the final version you want committed. The file must compile/run correctly with no leftover markers.

          

            
📄 app.py (resolved — no markers remain)

            

              greeting = "Hello from Alice and Bob!"

              print(greeting)
            

          

          

            ⚠️
            
The most common mistake is leaving a >>>>>>> or ======= line in the file accidentally. Always do a search for <<< in your editor before staging.

          

        

      


      

        
4c

        

          
Stage the resolved file with git add

          
Adding a conflicted file is how you signal to Git that you have resolved it. This is different from a regular add — it marks the conflict as settled.

          

            $ git add app.py
          

          

            💡
            
Run git status again now. Notice the file has moved from "Unmerged paths" to "Changes to be committed". That means Git accepts your resolution.

          

        

      


      

        
4d

        

          
Complete the merge by running git commit

          
Git will open your default editor pre-populated with a merge commit message. You can accept it as-is or customise it. This commit officially records that the two branches have been merged.

          

            $ git commit

            # editor opens with: "Merge branch 'feature/bob'"

            # save and close the editor

            [main j1k2l3m] Merge branch 'feature/bob'
          

        

      


      

        
4e

        

          
Verify the final state of the repository

          
Run the graph log to see the complete picture: two branches forking from the initial commit, then converging at the merge commit on main.

          

            $ git log --oneline --all --graph

            *   j1k2l3m (HEAD -> main) Merge branch 'feature/bob'
|\
| * 7g8h9i0 (feature/bob) feat: personalise greeting for Bob
* | d4e5f6a (feature/alice) feat: personalise greeting for Alice
|/
* a1b2c3d initial: add greeting script


            $ cat app.py

            greeting = "Hello from Alice and Bob!"

            print(greeting)
          

          

            ✓
            Your merge is complete. git status shows "nothing to commit, working tree clean" — the repo is in a healthy state.
          

        

      


    

  


  
  
Bonus challenge


  

    

      
★

      

        
Optional · Visual editor

        
Repeat the exercise using VS Code's merge editor

        
VS Code has a built-in three-pane conflict UI — learn to use it for production workflows.

      

      
›

    

    


      

        
★a

        

          
Set VS Code as your default Git editor and merge tool

          

            $ git config --global core.editor "code --wait"

            $ git config --global merge.tool vscode

            $ git config --global mergetool.vscode.cmd 'code --wait $MERGED'
          

        

      


      

        
★b

        

          
Re-create the conflict from scratch and open the merge editor

          
Repeat phases 1–3 in a fresh folder, then instead of manually editing the conflict markers, open VS Code. It will show you inline buttons: "Accept Current", "Accept Incoming", "Accept Both", or open the full three-pane merge editor.

          

            $ git mergetool

            # VS Code opens with the three-pane view
          

          

            💡
            
In VS Code, look for the "Resolve in Merge Editor" button at the top of the conflicted file. The three-pane view shows: Ours (left) | Result (centre) | Theirs (right). Click checkboxes to accept individual hunks.

          

        

      


      

        
★c

        

          
Intentionally conflict three files at once and practice resolving them in sequence

          
Real-world merges often affect multiple files. Create config.py, utils.py, and app.py, then have both branches edit all three files. Run the merge, then resolve each file one by one, staging each before moving to the next.

          

            $ git status

            # identify all files marked "both modified"

            $ git add config.py  # resolve one, stage it

            $ git add utils.py   # then the next

            $ git add app.py    # then the last

            $ git commit
